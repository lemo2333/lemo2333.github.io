Spectra Messaging Service — Core Platform

Oracle · Distributed Messaging Infrastructure

Overview

SMS is a cloud-native pub/sub messaging service built on top of OCI Streams (Oracle's managed Kafka-compatible service). It serves as the messaging backbone for Oracle Fusion Applications, abstracting stream provisioning, channel lifecycle management, and subscription routing behind a clean REST API.

The architecture evolved across four major versions: from a single-tenant Phase-1 design to a fully multi-tenant multiplexed system in v4, while maintaining backward compatibility throughout.

Key Components

  • Subscription Manager — control plane service managing channel lifecycle and subscription CRUD. Consumers create subscriptions; producers create channels. Tracks resource scale requirements.
  • Resource Manager — allocates and manages underlying OCI Stream scale units, interfacing with K8s Operator for dynamic provisioning.
  • K8s Operator — bridges the control plane to OCI Streams, handling stream creation, deletion, and scaling in response to resource manager signals.
  • Messaging Data Plane — CtrlDemux + DataDemux services handle actual publish/consume operations at runtime, routing messages between producers/consumers and the OCI Streams substrate.
  • SPECTRA PUB/SUB Connector Lib — client-side library for producers and consumers to interact with SMS Open API.

Stream Multiplexing Engine (v4)

Oracle · Spectra Messaging Service

Problem

In early SMS versions, each logical channel mapped 1:1 to a physical OCI Stream. This was wasteful — OCI Streams have fixed cost regardless of utilization. As tenants scaled, we needed to multiplex multiple channels onto shared streams while ensuring fair bandwidth allocation and preventing noisy-neighbor issues.

4-Phase Multiplexing Algorithm

  • Phase 1 — DB Lookup: Retrieve existing stream pool sorted by allocated bandwidth (ascending).
  • Phase 2 — Sorted Pool Traversal: Walk streams from least-to-most allocated; find candidates below capacity threshold.
  • Phase 3 — Quick Capacity Check: O(1) early exit — if least-allocated stream can't fit the request, reject immediately without expensive metrics calls.
  • Phase 4 — Real-Time Metrics Analysis: Dual-threshold formula: utilization = (allocated × 0.5) + actual_OCI_throughput. Both allocated and measured utilization must stay below 80% for a stream to be reused.

Architecture Simplification

  • Merged Resource Manager into Subscription Manager — removed a gRPC hop from the critical path (REST API → ChannelService → gRPC → ResourceManager → OCI became REST API → ChannelService → OCI SDK directly).
  • Error handling now uses direct OCI error responses instead of translated gRPC status codes — simpler debugging.
  • Virtual Partitioning: each physical partition expands to 3ⁿ virtual partitions, enabling fine-grained per-tenant throughput scaling without provisioning new streams.

Object Storage Auto-Rehydration

Oracle · Spectra Messaging Service — 26.07 Release

Problem

The SMS control plane stores critical mapping data — the relationship between logical channels and physical OCI Stream OCIDs — exclusively in object storage. The FADB (Fusion Application Database) maintains channel attributes, but OCID mapping lives only in object storage buckets.

When a bucket is accidentally deleted (operator error, regional DR failover, or data corruption), the control plane fails catastrophically while the data plane continues operating unaffected. Without auto-rehydration, recovering this state required lengthy manual intervention and risked data loss.

Solution Design

  • Marker file (db.marker): On startup, CtrlDemux checks for a marker file in object storage. If missing, it signals a RocksDB reset / bucket loss and triggers the rehydration pipeline automatically.
  • Physical channel rehydration: Reads SMS_MT_CHANNEL_DEFS from FADB to get CHANNEL_ID, NAME, RETENTION, PARTITIONS; uses ADDITIONAL_DETAILS to retrieve OCI Stream OCIDs; pushes mapping back to object storage.
  • Consumer table rehydration: Reads SMS_MT_Consumers FADB table to get subscriptionId, consumerId, batchSize, offset; calls rehydration gRPC endpoint on CtrlDemux to restore object storage consumer state.
  • Failover / Fallback handling: DemuxBatchObserver in DataDemux listens for DR events and triggers per-subscription or batch rehydration via the rehydration endpoint.

API Rate Limiting Framework

Oracle · Spectra Messaging Service

Problem

SMS operates as a shared multi-tenant service. A single tenant making excessive control plane API calls (channel creation, subscription management, etc.) could degrade performance for all other tenants. No rate-limit enforcement existed at the API layer.

Architecture

  • Entry Point: Traefik ingress / API Gateway applies blanket rate limits per ingress route.
  • Rate Limiter Interceptor (ContainerRequestFilter): Intercepts every request per replica. Parses HTTP method, extracts URI path params (customer-id, channel, subscription). Determines applicable rate-limit policy. Computes rate-limit key and checks/consumes tokens from cache.
  • Policy Store: YAML config loaded via Helm values. Scope key = customer + apiGroup(/channel/|/subscription) + method. Defines limits (requests_per_second, burst). Supports path-prefix matching and method-specific overrides.
  • Caffeine Cache (Token Bucket Backend): Low-latency in-process cache with 300s TTL for atomic token increments. Returns ALLOW (remaining tokens) or DENY (429 Too Many Requests with retry-after header).
  • Observability: Metrics per customer/path/method/status. Alerts on spikes, top offenders, backend latency. Audit logs with correlation IDs.

NACK — Message Redelivery Support

Oracle · Spectra Messaging Service

Problem

SMS consumers previously had no way to signal message processing failures. If a consumer crashed mid-batch or failed to process a message group, those messages were effectively lost — there was no redelivery mechanism. This violated at-least-once delivery guarantees required by upstream Fusion Applications.

Design

  • Handling GetMessage Requests: When DataDemux delivers a batch to a consumer, it saves the last non-committed batch offset into a checkpoint table in RocksDB at delivery time.
  • Handling NACK Requests: On receiving a NACK, DataDemux creates a gRPC request to CtrlDemux to fetch the last non-committed batch from RocksDB for that consumer, then delivers it back as the response to the next getMsg call.
  • MsgAsyncQueue: A demuxBatch work item is queued when NACK arrives. prevBatch = deliveredBatch is set so the next getMsg sequence returns prevBatch instead of advancing the offset.
  • internalAck change: internalAck is now streamed after (not before) the Msgs response so NACK semantics are consistent — consumers can NACK a batch they have already received but not yet processed.
  • Offset Semantics: Any sequence of getMsg calls should return currentBatch. Only on a successful ack does the offset advance.

Self-Recovery Monitoring Service

Oracle · Spectra Messaging Service

Problem

SMS cluster failures (stream loss, data corruption, service restarts) previously required manual operator intervention. Each failure mode had its own ad-hoc runbook. There was no unified framework to detect anomalies and automatically execute recovery steps, leading to extended downtime and operational burden.

Design Principles

  • False positives are strictly avoided — recovery actions only trigger on confirmed anomalies.
  • Recovery actions move through a code-based state machine (pipeline) to prevent multiple simultaneous actions.
  • State is persisted to FADB so recovery can be restarted if the monitoring service itself fails.
  • Pluggable framework — simple interfaces for adding new recovery actions and events without modifying core logic.

9-Stage Recovery Flow

  • Stage 1–2: Scale down data-plane services (DataDemux, CtrlDemux); detect stream loss and trigger OSS stream re-provisioning.
  • Stage 3–4: Sub-Mgr cleans up abstract channels without partitions; Res-Mgr repopulates subscriptions into Object Storage.
  • Stage 5–6: Subscription cleanup (synchronous); Sub-Mgr retrieves subscription list from FADB and passes to Res-Mgr.
  • Stage 7: Consumer-side cleanup — Data-Demux rehydrates Metadata RocksDB, recalculates offsets in FADB and RocksDB, invalidates consumers to decoupled state.
  • Stage 8: Restart API; confirm Kubernetes readiness check.
  • Stage 9: Declare recovery success, halt state machine.
