Oracle · Spectra Messaging Service

Spectra Messaging Service — Core Platform

Problem Statement

Oracle Fusion Applications needed a unified pub/sub messaging backbone for inter-service communication across thousands of tenants. Each team integrating directly with OCI Streams duplicated auth, stream provisioning, retry logic, and SLA enforcement. We built SMS as a managed messaging layer that abstracts all substrate complexity behind a clean channel-subscription REST API.

Solution Principles

  • 1

    Separation of control and data planes

    Control plane manages channel/subscription lifecycle and stream allocation. Data plane (CtrlDemux + DataDemux) handles runtime message flow independently — failures in one don't cascade to the other.

  • 2

    Substrate abstraction via K8s Operator

    Producers and consumers never interact with OCI Streams directly. The K8s Operator provisions and scales streams transparently in response to Resource Manager signals.

  • 3

    Logical channels over physical streams

    A logical channel maps to one or more OCI Stream partitions. In later versions (v2–v4), multiple channels are multiplexed onto shared streams, dramatically reducing cost.

  • 4

    SPECTRA connector library

    Client-side SDK abstracts gRPC messaging operations, internal ack semantics, and retry logic so application teams write zero boilerplate.

Architecture

SMS high-level architecture — Control Plane manages setup; Data Plane handles runtime message flow

Concrete Example

A Fusion Payroll service wants to notify downstream modules when payroll is processed. The Payroll service (producer) calls POST /v1/channel → SubMgr creates a logical channel, ResMgr allocates partition capacity on OCI Streams, K8s Operator provisions the stream. The GL Accounting service (consumer) calls POST /v1/subscription. At runtime, Payroll publishes via the SPECTRA lib → DataDemux fans out to the OCI Stream → DataDemux delivers batches to GL Accounting on poll.

Your Role

Designed and implemented the Subscription Manager service (channel/subscription lifecycle APIs), contributed to Resource Manager stream-allocation logic, and led the architecture evolution from single-tenant Phase-1 to the multi-tenant multiplexed design across v2–v4. Also owned the DataDemux consumer-routing layer and SPECTRA connector library integration patterns.

Future Directions

  • Cross-region replication of channel metadata to support active-active deployments
  • Consumer group semantics (Kafka-style group offsets) to allow competing-consumer patterns without client coordination
  • Observability dashboard exposing per-channel throughput, lag, and SLO burn rate to tenant teams

FAQ

Why not let teams use OCI Streams directly?

Direct usage means each team manages IAM policies, stream provisioning, retention config, and DR themselves. SMS centralizes this, enforces SLOs uniformly, and enables features like multiplexing and rehydration that are impractical to build per-team.

How is SMS different from Kafka?

SMS runs on top of OCI Streams (a Kafka-compatible managed service) and adds a higher-level abstraction: logical channels with named subscriptions, tenant-aware routing, and automatic stream lifecycle management. Clients see a REST + gRPC API, not raw Kafka protocol.

What happens when OCI Streams is unavailable?

The data plane degrades gracefully — publishes queue in-memory with backpressure signaling. The Self-Recovery Monitoring Service detects stream loss and triggers automated remediation (see that project card).

Oracle · Spectra Messaging Service — v4 Architecture

Stream Multiplexing Engine

Problem Statement

In SMS v1–v2, each logical channel mapped 1:1 to a physical OCI Stream partition. OCI Streams are billed per-stream regardless of utilization, so a tenant with 100 low-traffic channels paid for 100 streams. As Fusion Applications scaled to thousands of channels across hundreds of tenants, stream costs exploded and OCI quota limits became a hard constraint. We needed to multiplex multiple channels onto shared physical streams without sacrificing isolation or throughput guarantees.

Solution Principles

  • 1

    4-phase stream selection algorithm

    DB lookup → sorted pool traversal → O(1) quick capacity check → real-time dual-threshold analysis. Each phase is a progressively more expensive gate, so most decisions exit early.

  • 2

    Dual-threshold utilization formula

    utilization = (allocated_bandwidth × 0.5) + actual_OCI_throughput. Both the calculated score and the measured utilization must stay below 80% for a stream to accept a new channel. This prevents both over-booking and hotspot formation.

  • 3

    Merge Resource Manager into Subscription Manager

    Removed the gRPC hop between SubMgr and ResMgr. Channel creation now calls OCI SDK directly: REST → ChannelService → OCI SDK. Simpler error handling, lower latency, no serialization overhead.

  • 4

    Virtual partitioning (3ⁿ expansion)

    Each physical partition expands to 3ⁿ virtual partitions. A tenant needing more throughput re-partitions without provisioning new streams. The multiplexing algorithm re-maps virtual → physical transparently.

Architecture

Top: before vs. after multiplexing · Bottom: 4-phase stream selection algorithm

Concrete Example

Tenant Pepsi has channels A, B, and C each requesting 200 KB/s. Stream-1 has 1 MB/s capacity and is already at 400 KB/s allocated. Phase ①: DB returns [Stream-1 (400KB), Stream-2 (900KB)]. Phase ②: Stream-1 is the candidate. Phase ③: 400+200=600 < 1000 KB/s — quick check passes. Phase ④: actual OCI throughput on Stream-1 is 350 KB/s; score = (400×0.5)+350 = 550, utilization = 55% < 80% → ALLOCATE Channel C onto Stream-1.

Your Role

Designed the 4-phase selection algorithm from scratch, including the dual-threshold utilization formula and the O(1) early-exit optimization. Led the architectural decision to merge Resource Manager into Subscription Manager, eliminating the gRPC hop. Implemented the virtual partitioning expansion (3ⁿ) and the channel-to-stream mapping update API in Phase-2.

Future Directions

  • Predictive pre-warming — use historical channel growth patterns to pre-allocate stream capacity before quota exhaustion
  • Per-tenant SLA tiers — premium tenants get dedicated streams; standard tenants share multiplexed streams
  • Dynamic rebalancing — periodically re-evaluate channel-to-stream mappings to correct imbalances that accumulate over time

FAQ

How do you prevent one noisy tenant from starving others on a shared stream?

The dual-threshold formula uses actual OCI throughput — if a tenant bursts, the measured utilization rises above 80% and the stream becomes ineligible for new channels. Per-tenant write-bandwidth alerts trigger before sustained overuse reaches the shared stream's saturation point.

What happens to in-flight messages during virtual partition expansion?

The channel must be deactivated before partition count changes — consumers drain existing messages, then the channel is reactivated with the new partition mapping. The Update Channel API (Phase-2) enforces this lifecycle gate.

Oracle · Spectra Messaging Service — 26.07 Release

Object Storage Auto-Rehydration

Problem Statement

The SMS control plane stores the critical mapping between logical channels and physical OCI Stream OCIDs exclusively in object storage buckets. When a bucket is accidentally deleted — through operator error, regional DR failover, or data corruption — the control plane fails catastrophically: channel creation, publish, and consume all break. The data plane continues unaffected (it uses a config map, not object storage), creating an asymmetric outage that is hard to diagnose. Without auto-rehydration, recovery required hours of manual intervention and risked permanent data loss.

Solution Principles

  • 1

    Marker-file sentinel for reset detection

    On startup, CtrlDemux writes a db.marker file to object storage containing the RocksDB instance ID. If the file is absent on next startup, a bucket loss or RocksDB reset is confirmed and rehydration triggers automatically.

  • 2

    FADB as the source of truth

    The Fusion Application Database (FADB) holds channel attributes, partition config, and consumer subscription state. Rehydration reads from FADB and pushes the reconstructed mapping back to object storage — no manual data entry required.

  • 3

    Per-subscription and batch rehydration paths

    Consumer table rehydration runs per-subscription (for targeted repair) or in batch (for full bucket recovery). The DataDemux DR event listener triggers batch rehydration on failover events from OCI DR tooling.

  • 4

    Zero-intervention recovery

    The entire flow is fully automated. Operator action is only required to initiate a DR failover — the rehydration pipeline handles all downstream state reconstruction without tickets or runbooks.

Architecture

Three states: normal operation → bucket loss → auto-rehydration from FADB source of truth

Concrete Example

During a quarterly DR drill, the OCI object storage bucket for the SMS control plane in region PHX is deleted as part of the failover procedure. CtrlDemux in the standby region (ASH) starts up and checks for db.marker — it's absent. The startup rehydration pipeline fires: it reads all rows from SMS_MT_CHANNEL_DEFS in FADB (channel name, OCID, retention, partition count), reconstructs the physical channel mapping JSON, and writes it to a new bucket in ASH. It then reads SMS_MT_Consumers to rebuild subscription-to-offset mappings. Total recovery time: ~90 seconds, zero operator intervention.

Your Role

Designed the marker-file sentinel pattern and the full rehydration pipeline architecture. Implemented physical channel rehydration (FADB read → object storage write) and the consumer table rehydration gRPC endpoint on CtrlDemux. Defined the three trigger modes (startup, DR event, manual API) and wrote the runbook that became the automated recovery flow.

Future Directions

  • Periodic bucket integrity checks — proactively detect partial corruption before a full outage occurs
  • Cross-region bucket replication as a faster recovery path than full rehydration from FADB
  • Rehydration progress API so operators can monitor recovery status in real-time during DR events

FAQ

Why store the OCID mapping only in object storage and not FADB?

FADB is the FA governance database — it's not designed for high-frequency read access by the data plane. Object storage provides fast, durable key-value lookups at the read volumes the control plane needs. FADB remains the source of truth for channel configuration, feeding the rehydration pipeline when needed.

Can rehydration cause duplicate message delivery?

No. Rehydration only restores the channel-to-stream mapping and consumer subscription metadata. The actual message data lives in OCI Streams partitions, which are unaffected by bucket loss. After rehydration, consumers resume from their last committed offset stored in RocksDB, not from the beginning.

API Rate Limiting Framework

Problem Statement

SMS is a shared multi-tenant service. A single tenant making excessive control plane API calls — channel creation bursts, subscription polling, or misconfigured SDK retry storms — could exhaust thread pools and degrade performance for all other tenants. No rate-limit enforcement existed at the API layer; the only protection was Kubernetes HPA, which reacts too slowly to traffic spikes. We needed per-customer, per-endpoint enforcement with sub-millisecond overhead and zero coordination between replicas.

Solution Principles

  • 1

    Intercept at the application boundary

    JAX-RS ContainerRequestFilter runs before every request on every replica. It extracts customer-id, channel name, and subscription ID from the URI path, then determines which rate-limit policy applies.

  • 2

    Token bucket via in-process Caffeine cache

    Each rate-limit key (customer + api-group + method) maps to a token bucket stored in a Caffeine cache (TTL 300s). No Redis, no network round-trip — atomic increments stay in-process for sub-millisecond decisions.

  • 3

    Declarative policy store

    Policies are defined in YAML under Helm values: scope (customer + path prefix + method), requests_per_second, and burst. Operators can tune limits per customer tier without code changes — just a Helm upgrade.

  • 4

    Graceful signaling on denial

    On DENY, the filter returns HTTP 429 with Retry-After and X-RateLimit-Reset headers so SDKs can back off correctly. This prevents retry storms from making the problem worse.

Architecture

Request flow through 4 layers: Ingress → ContainerRequestFilter → Policy + Token Bucket → Allow/Deny

Concrete Example

Tenant "Pepsi" has a misconfigured SDK that retries channel-get every 10ms. The policy for customer=pepsi + /channel/ + GET allows 10 rps with burst 20. After 20 requests hit Caffeine's token bucket in rapid succession, tokens are exhausted. The ContainerRequestFilter returns 429 with Retry-After: 8s. The SDK backs off. Meanwhile, other tenants on the same replica are completely unaffected — their buckets are independent keys in the Caffeine cache.

Your Role

Designed the ContainerRequestFilter interception pattern, the rate-limit key schema (customer + apiGroup + method), and the Caffeine-backed token bucket implementation. Defined the YAML policy store schema under Helm values and implemented path-prefix matching with method-specific override support. Set up the p95/p99 alerting pipeline for top offenders and backend latency.

Future Directions

  • Distributed rate limiting via a shared Coherence/OCI Cache cluster to enforce limits across all replicas simultaneously (current per-replica buckets allow N-replica burst)
  • Dynamic policy updates without Helm redeploy — watch a ConfigMap and reload limits at runtime
  • Graduated throttling — slow down requests progressively (e.g. add artificial delay) before hard-denying to improve SDK behavior

FAQ

Why Caffeine instead of Redis for the token bucket?

Redis adds a network round-trip (1–5ms) per request, which is significant overhead for a filter that runs on every API call. Caffeine is in-process with nanosecond atomic operations. The tradeoff is that limits are enforced per-replica rather than globally — a tenant on 3 replicas can burst at 3× the stated limit. This is acceptable given our replicas handle ~1/3 of traffic each; we document this behavior and plan distributed enforcement as a future upgrade.

What's the performance impact on the normal request path?

Measured overhead is <0.1ms per request — dominated by URI path parsing, not the Caffeine lookup. The filter exits early on ALLOW without any additional I/O. We verified no measurable impact on p99 latency in load tests at 2,000 rps.

NACK — Message Redelivery

Problem Statement

SMS consumers fetch message batches from DataDemux via gRPC. If a consumer processes a batch and crashes mid-way — or explicitly detects a business-logic failure — those messages were silently lost. There was no mechanism to signal "I did not process this batch, please redeliver." Upstream Fusion Applications require at-least-once delivery guarantees; without NACK support, engineers worked around this with expensive application-level idempotency layers that duplicated offset tracking.

Solution Principles

  • 1

    Checkpoint at delivery, not acknowledgement

    When DataDemux delivers a batch, it saves the batch offset to a checkpoint table in RocksDB before returning the response. This ensures the "last delivered batch" is always recoverable, even if the consumer crashes immediately after receiving it.

  • 2

    NACK triggers RocksDB lookup and redeliver

    On receiving a NACK, DataDemux fetches the last non-committed batch from the RocksDB checkpoint table for that consumer and returns it as the response to the next getMsg call — no round-trip to OCI Streams required.

  • 3

    internalAck timing change

    Previously, internalAck was streamed before the Msgs response (optimistic). NACK semantics require internalAck to be sent after the consumer receives the batch — otherwise a mid-delivery crash would advance the internal offset and make NACK impossible.

  • 4

    MsgAsyncQueue work item

    On NACK, a demuxBatch work item is enqueued in MsgAsyncQueue with prevBatch = deliveredBatch. The next getMsg call dequeues this item and returns prevBatch, providing a seamless redelivery experience to the consumer.

Architecture

Left: normal ACK path — offset advances. Right: NACK path — batch N redelivered from RocksDB checkpoint.

Concrete Example

The GL Accounting service fetches Batch N (10 payroll messages). After processing 7 messages, a downstream DB connection fails. The consumer calls NACK. DataDemux sees prevBatch=N in MsgAsyncQueue. The next getMsg response returns Batch N again from the RocksDB checkpoint. The consumer retries, completes successfully, and sends ACK. DataDemux updates the offset to N+1. The 3 messages that were already processed are re-processed — consumers must be idempotent, but no messages are lost.

Your Role

Designed the NACK semantics end-to-end: checkpoint-at-delivery pattern, RocksDB lookup on NACK, MsgAsyncQueue work item structure, and the internalAck timing fix. Implemented the DataDemux changes (checkpoint write, NACK handler, prevBatch dequeue logic) and the gRPC API contract changes for the NACK signal.

Future Directions

  • Dead-letter queue — after N consecutive NACKs on the same batch, route to a DLQ so poison messages don't block the consumer indefinitely
  • NACK with delay — allow consumers to request redelivery after a configurable backoff window (e.g. "retry in 30s") for transient failure handling
  • NACK metrics — track NACK rate per subscription to surface struggling consumers before they cause SLO breaches

FAQ

How does NACK interact with multiple consumers on the same subscription?

Each consumer instance has an independent RocksDB checkpoint entry keyed by consumer-id. A NACK from consumer instance A only redelivers to A — it doesn't affect other consumer instances polling the same subscription. Each instance manages its own offset independently.

What if the consumer NACKs forever (buggy consumer)?

Currently, redelivery is unbounded — this is intentional since the alternative (data loss) is worse for Fusion Applications. The dead-letter queue (planned future work) will cap retries and route stuck batches for manual inspection.

Self-Recovery Monitoring Service

Problem Statement

SMS cluster failures — OCI Stream loss, RocksDB corruption, service crashes — each had their own ad-hoc runbook requiring manual operator intervention. Recovery steps were error-prone, took 30–90 minutes, and required deep system knowledge. There was no unified framework to (a) detect anomalies early, (b) prevent simultaneous conflicting recovery actions, and (c) resume recovery if the monitoring service itself restarted mid-flow.

Solution Principles

  • 1

    Code-based state machine for safe sequencing

    Recovery actions execute through a typed state machine pipeline. Only one recovery flow runs at a time. The current stage is persisted to FADB so recovery can resume from the correct stage after a monitoring service restart — not from the beginning.

  • 2

    False-positive avoidance

    Before triggering recovery, the monitoring service verifies the anomaly from multiple signals (OCI metrics, K8s liveness, gRPC health checks). A single transient spike never triggers recovery — only sustained, confirmed failure states do.

  • 3

    Pluggable event and action framework

    New failure modes are added by implementing a simple Event interface and attaching Action implementations to state transitions. No core state machine changes required — new scenarios extend the framework without risking existing recovery flows.

  • 4

    Kubernetes readiness gate

    Recovery is only declared successful after all SMS pods pass the K8s readiness check. This ensures the control plane is actually serving traffic before alerting are cleared — not just that the recovery steps completed.

Architecture

Top: Monitoring Service polls all SMS components. Middle: State machine with FADB persistence. Bottom: 9-stage recovery pipeline.

Concrete Example

At 2am, OCI Streams metrics show the stream backing tenant Pepsi's channels has zero throughput for 10 minutes. The Monitoring Service confirms: DataDemux gRPC health returns UNAVAILABLE, K8s liveness fails on 2 of 3 CtrlDemux pods. Stage ① scales down DataDemux and CtrlDemux. Stage ② detects stream loss, provisions a replacement OCI Stream via the K8s Operator. Stages ③–⑦ clean up stale state and rehydrate RocksDB. Stage ⑧ scales services back up; K8s readiness passes on all pods within 45 seconds. Stage ⑨ clears the alert and sends a Slack notification to on-call. Total downtime: ~4 minutes vs. 60+ minutes with manual runbooks.

Your Role

Designed the state machine architecture and the FADB persistence contract for recovery stage resumability. Defined the pluggable Event/Action interface that allows new failure scenarios to be added without modifying the core pipeline. Implemented stages ③ (channel cleanup), ④–⑤ (subscription rehydration), and ⑨ (recovery declaration + alerting). Owned the false-positive avoidance logic using multi-signal confirmation.

Future Directions

  • ML-based anomaly detection to catch subtle degradation patterns (rising p99 latency, increasing NACK rates) before they become full outages
  • Automated chaos testing — weekly scheduled runs of the recovery pipeline against a staging cluster to validate each stage stays correct as the codebase evolves
  • Recovery playbook UI — real-time dashboard showing current stage, estimated completion, and manual override controls for operators who want to step through recovery interactively

FAQ

What if the Monitoring Service itself crashes mid-recovery?

On restart, the Monitoring Service reads the last persisted recovery stage from FADB and resumes from that stage. Each stage is idempotent — re-running a completed stage is safe. The K8s liveness check on the Monitoring Service itself fails if recovery state is corrupt, triggering a pod restart that re-enters the pipeline cleanly.

How do you prevent the recovery from making things worse?

Stage ① deliberately scales down the affected services before any repair actions. This stops the bleeding and ensures repair steps operate on quiescent state — no in-flight requests racing with recovery logic. Stages execute sequentially; each must complete successfully before the next begins.
